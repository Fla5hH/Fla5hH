# 《漏洞挖掘》

### 1、什么是SQL注入？

SQL 注入是一种将恶意 SQL 语句插入到输入参数中，从而操纵后端数据库执行非预期查询的注入攻击。

其根本原因在于，应用将用户输入与 SQL 语句进行了直接拼接，且未做有效的过滤或参数化处理。

攻击者可以利用它绕过认证、窃取敏感数据、篡改数据库记录甚至控制服务器。

在 OWASP Top 10（2021版）中，它属于 A03 注入大类。

防御手段核心是使用参数化查询、输入校验和最小权限原则。

### 2、SQL注入的两种基本类型

##### ①数字型注入

假设存在URL为 https://www.aaa.com/test.php?id=1

可以猜测后端语句为
    
    
    SELECT * FROM table WHERE id=1

参数位置为id，可以采用数字型注入进行判断
    
    
    SELECT * FROM table WHERE id=1'

输入单引号不符合语法，所以会导致页面报错
    
    
    SELECT * FROM table WHERE id=1 and 1=1

语句正常，页面无异常继续执行
    
    
    SELECT * FROM table WHERE id=1 and 1=2

还可以用运算符判断
    
    
    SELECT * FROM table WHERE id=1/0

如果除以0报错 但是除以1不报错 即为数字型

以上方法可大致判断出存在数字型注入

###### ②字符型注入

猜测后台SQL语句如下
    
    
    SELECT * FROM table WHERE username = 'admin'

采用字符型注入进行判断
    
    
    SELECT * FROM table WHERE username = 'admin''

输入admin'导致闭合错误，页面报错
    
    
    SELECT * FROM table WHERE username = 'admin' and 1=1 --+'

输入admin' and 1=1 -- 页面回显正常 admin' and 1=1 -- -也可以

\--为数据库中的注释 目的是为了注释掉原本参数里的单引号 使得and 1=1可以逃逸出原本的SQL语句

+在URL编码中 与 空格 的编码一致 可以绕过一些拦截 或者用/**/注释代替空格也可以
    
    
    SELECT * FROM table WHERE username = 'admin' and 1=2 --+'

判断条件为假，页面报错，通过以上测试可以初步判断存在字符型注入

###### ③其他注入类型

  * POST注入：注入字段在 POST 数据中
  * Cookie注入：注入字段在 Cookie 数据中
  * 延时注入：使用数据库延时特性注入
  * 搜索注入：注入处为搜索的地方
  * base64注入：注入字符串需要经过 base64 加密



底层原理都是数字型或者字符型注入
